Here are some stats from running it on my laptop:

Sorting Algo Runtime 10,000 items 100,000 items 100,000,000 items
Bubble 0.324s 35.58s Too long
Selection 0.127s 12.56s Too long

I probably wasn't timing these in the best way, but it still gives a good overall look at comparing O(n^2) and O(nlgn) runtimes. I think sorting algorithms are really interesting intro level algorithms that give a taste for what computer science is all about.

I would like to eventually go back and add some more sorts to this list as well as improve my implementations of the current ones. WEB DEV WOES

Some reasons why I will never be a web developer, August 19, 2022

To begin, I'm thinking about adding some kind of 'boring' tag whenever I get a tag system working. A lot of these notebook entries will probably end up being more of a changelog than actual fun or informative posts. For now, however, they will remain in one big list. Also note that if you read about a feature on an old post, it is very likely to have been changed in some way in the future.

I finished up that little spinny thing at the top. It was all fine and good yesterday, but then I decided I wanted to fix the animation weirdness and ran into a whole slew of issues. For some reason, adding a smooth transition in CSS alone would not fix the width of the box changing abruptly. It required that I also manually change the width in code rather than letting the browser handle it automagically. This was good because it meant that I could size it just how I wanted it (i.e. make it fit the phrase it landed on rather than the longest random phrase that was generated), but it meant that I would have to figure out how long to actually make it. This can be a difficult problem since most fonts are not monospaced, meaning that each letter takes up the same amount of pixels.

I made the box use a monospaced font. Problem solved. Except that none of the smooth transitions worked on the first time it was clicked. This was because 'style.width' component of that div didn't actually exist until I set it for the first time. If there is no previous width, then there is no way to smoothly transition to the new width. I solved this by setting the width of the div to the width of the div (do you see why I dislike this yet?) which is stored in the 'offsetWidth' component (which, of course, is read-only and can't just be used to change the width) once the page was loaded in. Why can't the 'style.width' just have a value at the beginning? (It is because I didn't hardcode in a width attribute at the start, I know, but why can't this value be whatever width the browser decided to give it?) To finish it off, I had to also wait until the page was loaded to add the smooth transition style to div. Otherwise, it would have this funky effect when the page was first loaded in.

I usually enjoy the code writing and debugging process. I feel that it is similar to solving a puzzle or investigating some kind of crime where I get to follow a trail of clues until I find the culprit. This is not the feeling that I get when working on websites. Instead, it is more like someone gave me an article to edit, asked me to update some of the wording using a thesaurus, and then yelled at every single suggestion that I ended up making. Instead of testing an algorithm to figure out the cases that are causing it to fail, I am trying seemingly random combinations of keywords until I end up finding what I want. Like with that 'style.width' versus 'offsetWidth' thing above. I have used 'style.width' in the past because that is how you change the width of something on the page dynamically, but now I had to search through random StackExchange posts to figure out that there is this whole separate width thing. There is no trail of breadcrumbs in the code for me to follow, just random Google searches to find the one page of documentation that can actually help me.

Okay, rant over. Idk if a lot of these posts will be ranty, but we will see. :)

Drowning in a sea of attribute names,
Denver
